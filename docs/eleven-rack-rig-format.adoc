= Avid/Digidesign Eleven Rack — rig file format (.tfx / .rig)
:toc: left
:icons: font
:source-highlighter: rouge

The on-disk format of an Eleven Rack rig preset, as reverse-engineered from the
Eleven Rack User Group (ERUG) rig collection. This is the basis for importing
community/factory rigs into the planned `rackctl-eleven` library, and -- correlated
with the live SysEx address map (`eleven-rack-sysex-protocol.adoc`) -- for building
the parameter catalog.

NOTE: Eleven Rack, Digidesign and Avid are trademarks of Avid Technology, Inc.
This is an independent, unofficial project. No rig files are committed; only this
derived format description.

== Corpus

The format below was derived by diffing a corpus of *1,243 `.tfx` rigs* (the
"All ERUG Rigs as of 03/25/21" community collection). The corpus is *not* committed
(third-party content, public repo); it is git-ignored under `/11/`. To reproduce,
place a set of `.tfx` files there.

== It is binary, not XML

A common community claim is that rig files are "XML if you rename them to `.xml`".
That is *not* true of these `.tfx` files -- they are a Digidesign *binary* format
(magic `DigiElvRELVhRig`). The XML claim may apply to a different export path; treat
it as false for `.tfx` until a real XML sample appears.

== Word orientation

The file is an array of *32-bit little-endian words*. Consequently four-character
tags and embedded text read *byte-swapped per 4-byte group* when scanned linearly:

[cols="2,2,3",options="header"]
|===
| On disk (raw) | Logical | Meaning
| `1loV` | `Vol1` | volume 1
| `opmT` | `Tmpo` | tempo
| `1cXF` | `FXc1` | FX-chain slot 1
| `.rMaelC..n` | `Mr. Clean` | rig name (text, same word-swap)
|===

The magic at the top reads as literal ASCII; the body (name, tags, values) is
word-swapped. A parser reads u32 LE and swaps tag bytes to recover the FourCC.

== Layout

[cols="1,3",options="header"]
|===
| Region | Contents
| word 0 | total file size in bytes, *big-endian* (e.g. `00 00 03 E8` = 1000)
| header | magic `DigiElvRELVhRig` (literal ASCII), then flag words
| name | the rig name as word-swapped text, from offset 64
| body | from offset 92: a sequence of *blocks*, one per signal-chain slot
|===

=== Block structure (cracked; validated across all 1,243 rigs)

The body is a sequence of blocks. Each is a *header word* followed by its parameter
entries:

----
header word, little-endian bytes:  <size_lo> <size_hi> 0x20 <block_id>
   block_id : the signal-chain slot (fixed; see table)
   0x20     : constant marker
   size     : payload size in bytes = (number of params) * 8
then size/8 entries, each 8 bytes:  <FourCC tag, LE> <32-bit value>
----

Tags are 4 chars, possibly space-padded (`Vol `, `Min `). Every rig carries the same
11 blocks:

[cols="1,2,4",options="header"]
|===
| Block id | Slot | Example params
| `0x41` | Global (rig) | RVol, Vol1/2, RMno, Tmpo (µs/beat), Msyc, ExpT, PIGI, FXc1..4, Wor*/Wst* routing
| `0x43` | Volume pedal | bypa, `Vol `, `Min `, Tapr
| `0x44` | Wah | bypa, Filt, VxCr
| `0x45` | Distortion / stomp | bypa, Driv, Tone, Levl
| `0x46` | EQ | bypa, LwSh, LMGn, MGn, HMGn, HiSh, Out
| `0x47` | Modulation | bypa, Sped, Dpth, Fdbk, PDly, Sync
| `0x48` | Modulation 2 / tremolo | bypa, Sped, Sync
| `0x49` | *Amp* | sld1..sldO (the amp model's knobs as generic slider slots)
| `0x4A` | FX loop | bypa, send, rtrn, wetp
| `0x4B` | Delay | bypa, `Vol `, Sust, EDly, Sync, Rec, Wow, Tilt
| `0x4C` | Reverb | bypa, Dcay, Tone, RMix, Type, PDly
|===

NOTE: `.tfx` block ids are *fixed signal-chain slots* (amp is always `0x49`), whereas
the *live SysEx* block id is *model-dependent* (the amp's DC Modern model answered at
SysEx block `0x3D`). The two are parallel views, correlated via the knob sweeps.

=== Value encoding

Global-block (`0x41`) values are *integers* and decode directly: `Tmpo` = 500000
(µs/beat = 120 BPM), `PIGI` = 0..127, `Msyc`/`RMno` = 0/1, the `Wor*`/`Wst*` routing
are small ints. Effect-block values are *32-bit floats*; an unset/default parameter
holds a sentinel (`0x7FFFFFFF` or `0x80000000`). The amp's knobs live in block `0x49`
as `sld1..sldO`; which slider is which knob is model-specific, matched from the live
SysEx sweep + the manual's per-model knob list.

NOTE: the *same two sentinels* appear in the global block too (e.g. an unused
`RVol`/`Vol1`/`Vol2` reads `0x7FFFFFFF`/`0x80000000`), so "integers decode directly"
holds *except* for those unset words. The typed model (below) surfaces every unset
value — global or effect — uniformly as JSON `null`.

== Typed patch model

The library exposes a **typed, block-structured** patch
(`rackctl-eleven-model::patch::Patch`) built from a parsed `.tfx` file, grouped the
same way as the GX-700's `typed::Patch` so both products present a patch identically.
It is the on-disk form saved by `import` and printed by `import --json`:

[source,json]
----
{
  "name": "Marshall JCM 800",
  "global":      { "master_volume": null, "tempo_us": 497930, "input_gain": 5, ... },
  "distortion":  { "bypass": false, "drive": 0.75, "tone": null, "level": 0.5 },
  "amp":         { "sliders": { "sld1": 0.25, "sld2": null, ... } },
  "delay":       { "bypass": true, "time": 0.4, ... },
  "reverb":      { "bypass": false, "decay": 0.6, "mix": 0.3, ... }
}
----

Mapping rules, chosen to stay *honest about what is decoded*:

* Each of the eleven fixed blocks becomes a named struct (`global`, `volume_pedal`,
  `wah`, `distortion`, `eq`, `modulation`, `modulation2`, `amp`, `fx_loop`, `delay`,
  `reverb`); block order is the signal-chain order.
* The stable `FourCC` vocabulary is exposed as **named fields**; any other tag a
  patch carries is preserved verbatim under the block's `extra` map (or, for the
  global block, `routing`), so a round trip never drops a parameter.
* An **unset sentinel** decodes to `null` everywhere (`Option<i64>` for the global
  ints, `Option<f32>` for the effect floats).
* Effect values remain **raw 32-bit floats**: their scaling into display units
  (dB, %, ms) is the open item below, so the JSON is faithful-but-raw, not yet in
  human units. When that scaling is solved, only the field *values* change — the
  shape stays the same.

CAUTION: a `.tfx`-imported typed patch *cannot yet be written to the hardware*. The
`.tfx` block ids and float values are a different namespace from the live SysEx
address map (see below and `eleven-rack-sysex-protocol.adoc`); only a
device-*captured* `PatchBackup` can be restored to a slot today.

== Stable schema (key vocabulary)

~78 FourCC keys appear in ~99% of rigs -- the stable parameter vocabulary. Selected
keys with their *observed* value ranges and best-guess meaning (guesses flagged):

[cols="1,2,3",options="header"]
|===
| Key | Range (raw) | Meaning (✓ = clear, ? = guess)
| `RVol` | wide | master rig volume ?
| `Vol1` / `Vol2` | wide | volume controls (rig A/B?) ?
| `RMno` | 0..1 | mono ? (✓ boolean)
| `Tmpo` | large | tempo ?
| `Msyc` | 0..1 | MIDI sync on/off ?
| `ExpT` | 0..4 | expression-pedal target/type ?
| `PIGI` | 0..127 | input gain ? (✓ 7-bit range)
| `FXc1`..`FXc4` | packed | the four FX-chain slots (effect id + state) ?
| `Driv` | float-ish | amp/stomp drive ?
| `Levl` | float-ish | level ?
| `Tone` | float-ish | tone ?
| `Dcay` | float-ish | decay (reverb/delay) ?
| `Filt` | float-ish | filter ?
| `Type` | enum | effect type selector ?
| `PDly` | float-ish | pre-delay ?
| `Sync` / `Tapr` | varies | delay sync / taper ?
| `send` / `rtrn` / `wetp` / `bypa` | varies | FX-loop send / return / wet / bypass ?
| `sld1`..`sldO` | float-ish | ~24 generic parameter "sliders" (per-effect params) ?
| `Wor[B..L]` | 11..91 | routing-graph node (wire origin?) ?
| `Wst[B..L]` | 0..9 | routing-graph node (wire stage/state?) ?
|===

The `Wor*`/`Wst*` pairs and `FXc1..4` together describe the *signal routing* (which
effect occupies each chain slot and how blocks are wired) -- the Eleven Rack's
hallmark flexible rig graph.

== Correlation method (rig file ↔ live SysEx)

The two reference points reinforce each other:

. Load a known rig on the unit.
. Read its parameter addresses over SysEx (`eleven-rack-sysex-protocol.adoc`).
. Open the same rig's `.tfx` and read its keyed values.
. Match values -> bind each SysEx address to a `.tfx` key (a real name + range).

This labels the address map and pins value encodings far faster than knob-sweeping
every parameter, and the corpus supplies each parameter's range for free.

== Open questions (tracked work)

* The effect-block *float* value scaling (ranges, units) and the meaning of the
  `0x7FFFFFFF`/`0x80000000` sentinels.
* Which `sld1..sldO` slot is which knob, per amp model (matched from the live sweep).
* Decode `FXc1..4` (effect id + slot state) and the `Wor*`/`Wst*` routing graph.
* Relationship between `.tfx` and any `.rig` variant.
* Whether the stored effect floats equal the live SysEx 32-bit words.

(The body block structure and the global-block integer values are *solved* — see
*Block structure* above, validated across all 1,243 rigs.)
