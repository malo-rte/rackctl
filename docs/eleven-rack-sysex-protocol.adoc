= Avid/Digidesign Eleven Rack — MIDI System Exclusive protocol reference
:toc: left
:icons: font
:source-highlighter: rouge
:experimental:

This document describes how the Avid (Digidesign) Eleven Rack guitar amp/effects
processor is controlled over MIDI, as it will be implemented by the planned
`rackctl-eleven` crate. It records what the device actually does -- confirmed
against a real unit -- and is the reference for the parameter catalog.

Unlike the GX-700, there is *no* published MIDI implementation. Avid never
documented the protocol the standalone "Eleven Rack Editor" used, and the editor
was discontinued (it required Pro Tools and broke on modern 64-bit hosts). Every
fact below was *reverse-engineered from hardware* (a unit running firmware
`0157`, reached over ALSA raw MIDI) and, where useful, cross-checked against the
community's prior art (USB sniffing with Wireshark / MIDI Monitor, and Max/MSP and
Python scripts that map parameters to CC). The device is the final word.

NOTE: Eleven Rack, Digidesign and Avid are trademarks of Avid Technology, Inc.
This is an independent, unofficial project, not affiliated with Avid; the names
identify the hardware these tools control.

== At a glance

* The Eleven Rack is a *rig-based* guitar amp/effects modeller and an 8x8 USB 2.0
  audio interface. Editing is done entirely over MIDI System Exclusive; there is no
  ALSA mixer surface for the DSP parameters.
* Control is *address-mapped System Exclusive* -- the same _shape_ as the GX-700's
  Roland scheme, but with Digidesign framing. Every parameter lives at a multi-byte
  address; you read it with a request and the unit answers with a report.
* The unit does *not* volunteer its state. A rig change does not stream the new
  rig -- reads must be *polled*, one address (or a batch of requests) at a time.
* Turning a front-panel knob emits an unsolicited change report for that one
  parameter, which is how addresses are discovered and labelled.

== Transport (ALSA)

The unit enumerates as a USB-MIDI device (ALSA card, two raw-MIDI ports):

[cols="2,3",options="header"]
|===
| Port | Use
| `hw:<card>,0,0` "Eleven Rack Rig" | Editor / parameter SysEx. *This is the port.*
| `hw:<card>,0,1` "Eleven Rack External" | DIN MIDI thru / external pedal. Silent in testing.
|===

The Rig port is silent when idle; all traffic is request/response or change-driven.
This reuses the byte-level `rackctl-midi` link unchanged.

== USB topology

For reference, the device's full USB descriptor (`0dba:b011`, USB 2.0 high-speed,
`bDeviceClass 0xEF` multi-interface, one configuration, five interfaces). Parsed from
the kernel's config descriptor, including alternate settings:

[cols="1,2,1,1,1,1,3",options="header"]
|===
| Interface | Class | Alt | Endpoint | Type | maxpkt | Purpose
| IF0 | DFU `fe.01.00` | 0 | -- (EP0) | -- | -- | Firmware upgrade
| IF1 | vendor `ff.01.20` | 0 | -- (EP0) | -- | -- | Vendor control / management
| IF2 | *MIDIStreaming* `01.03` | 0 | `0x02` | Bulk OUT | 512 | *MIDI / SysEx out*
| | | | `0x82` | Bulk IN | 512 | *MIDI / SysEx in*
| IF3 | vendor `ff.02.20` | 0 / *1* | `0x03` | Isoc async OUT | 416 | Audio playback (host -> unit)
| IF4 | vendor `ff.02.20` | 0 / *1* | `0x83` | Isoc async IN | 416 | Audio capture (unit -> host)
|===

What this means for the project:

* *Control is IF2* -- class-compliant MIDIStreaming over bulk endpoints
  `0x02`/`0x82`. That is why `snd-usb-audio` binds it and exposes the ALSA raw-MIDI
  ports; all the SysEx in this document flows through here. Nothing proprietary is
  needed for control.
* *The 8x8 audio is not class-compliant.* The streams are plain *vendor-specific
  isochronous* endpoints (IF3/IF4 alt 1), with no USB-Audio `AudioStreaming`
  descriptors, so the kernel will not drive them as a sound card. This is why the
  Eleven Rack needs Avid's proprietary driver for audio on a host, and it is why
  the audio side is *out of scope* (see `eleven-rack-roadmap.adoc`).
* *IF0 (DFU)* is firmware flashing and *IF1 (vendor control)* is what the legacy
  editor used for some control-transfer traffic; neither is needed, since the editor
  protocol we target runs over MIDI/SysEx on IF2.

== Message framing

All device messages share one envelope:

----
F0  13  0B  0F  <opcode>  <address...>  <value...>  F7
│   │   │   │
│   │   │   └ device id   0x0F
│   │   └ model           0x0B  (Eleven Rack)
│   └ manufacturer ID     0x13  (Digidesign)
└ SysEx start                              ... F7 = SysEx end
----

CAUTION: `0x13` as a one-byte MIDI manufacturer ID is what this unit emits; it is
recorded here as an observed fact, not from a registry.

== Device identity

The unit answers the *Universal* Identity Request, which is the safe first probe:

[cols="1,3"]
|===
| Request (universal) | `F0 7E 7F 06 01 F7`
| Reply               | `F0 7E 0F 06 02 13 0B 00 01 00 30 31 35 37 F7`
|===

Decoded reply: device id `0x0F`; manufacturer `0x13`; family `0x000B` (LSB-first
`0B 00`); model `0x0001` (`01 00`); version = ASCII `"0157"` (`30 31 35 37`).

== Opcodes

[cols="1,2,4",options="header"]
|===
| Opcode | Name | Direction / meaning
| `0x01` | Read request | host -> unit: `F0 13 0B 0F 01 <addr> F7`. Ask for one address.
| `0x12` | Read reply | unit -> host: `F0 13 0B 0F 12 <addr> <value> F7`. Solicited answer to `0x01`.
| `0x02` | Change report | unit -> host: `F0 13 0B 0F 02 <addr> <value> F7`. Unsolicited, when a knob moves.
| `0x03` | Write (set) | *Unconfirmed.* Presumed host -> unit set; verify by set + read-back, then restore.
|===

.Confirmed read round-trip (amp Gain)
----
->  F0 13 0B 0F 01 11 21 0D F7
<-  F0 13 0B 0F 12 11 21 0D 6D 00 00 00 10 F7
----

Batching many `0x01` requests then collecting the `0x12` replies enumerates an
address range in one pass -- the basis for a `scan`/`dump` command.

== Addressing

The address is *multi-byte* (three bytes seen). It behaves like the GX-700's
section/block/parameter scheme: a leading block selector, then a contiguous array
of parameter offsets.

Confirmed so far (firmware `0157`):

[cols="2,3",options="header"]
|===
| Address | Meaning
| `11 21 0D` | Amp Gain (front-panel knob)
| `11 21 00`..`11 21 7F` | The amp block -- a contiguous 128-entry parameter array (all answered a read)
|===

WARNING: Only the amp block (`11 21 xx`) is mapped so far. The full block map
(cab, stomps, mod, delay, reverb, FX loop, master) is open work -- discovered by
sweeping front-panel controls and correlating with the `.tfx` rig schema
(see `eleven-rack-rig-format.adoc`).

== Value encoding

Each value is *five MIDI bytes*: a 32-bit word packed as five little-endian 7-bit
groups (`b0` = least-significant 7 bits). Reconstruction:

[source]
----
value = b0 | (b1 << 7) | (b2 << 14) | (b3 << 21) | (b4 << 28)
----

Sanity checks from a live amp-block scan:

[cols="2,2,2",options="header"]
|===
| Address | Bytes | Decoded
| `11 21 00` | `40 00 00 00 00` | `0x40` = 64
| `11 21 02` | `7F 7F 7F 7F 0F` | `0xFFFFFFFF`
| `11 21 0D` | `6D 00 00 00 10` | low 7 bits = `0x6D` (Gain), plus a high tag bit
|===

What the 32-bit word *means* per parameter is *not yet uniform*: some parameters
are small integers (`0..127`, e.g. a knob position in the low 7 bits with a high
tag bit set), others appear to be 32-bit floats or packed multi-field words. A
naive IEEE-754 read of the whole word yields NaN, so the encoding is *not* a plain
float in this bit order.

IMPORTANT: Pinning each parameter's type, range and units is open reverse-engineering
work. The method: turn one knob through min / centre / max, capture the reports, and
correlate the same rig's stored values from its `.tfx` file
(see `eleven-rack-rig-format.adoc`).

== Rig selection

Front-panel rig changes do *not* emit a full parameter dump. Whether a rig is
selected by MIDI Program Change (as foot controllers like the FCB1010 / Ground
Control Pro use) and whether a whole-rig bulk dump opcode exists are *open
questions* -- to be probed while a unit is connected.
