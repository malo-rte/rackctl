= Avid/Digidesign Eleven Rack — audio-session driver: feasibility scope

Scoping study for a Linux userspace driver that activates the Eleven Rack's
proprietary USB audio session. Written 2026-07-02 from live-device descriptors
(`0dba:b011`) and the editor USB captures (`11/eleven-save-*.pcapng`).

== Status: the original motivation (editing) is MOOT — editing needs no audio

CAUTION (2026-07-02): this doc was written believing host parameter edits were gated
on an active audio session. *That was wrong* — host-set works over plain MIDI with the
correct block address (see `eleven-rack-sysex-protocol.adoc`, "host parameter set
WORKS"). The audio session is NOT required to edit. So this project is now only about:

* *Audio monitoring.* Give Linux users the Eleven's USB audio (guitar in / reamp /
  monitor), which no current driver provides — nice-to-have, no longer a blocker for
  the GUI editor.

The libusb activator + ISO streamer (`eleven-audio-probe`) already built for the
disproven edit-gate test are the reusable starting point if/when Linux audio is
pursued. The rest of this doc (USB topology, format, H1-H3 gate) stands as audio RE
reference.

== KEY FINDING (2026-07-15): the audio path is standard UAC2 wearing a vendor-class hat

Full decode of the latest cold-connect capture (`11/eleven-save-20260702-114348.pcapng`,
device address 108) settles every open question below. The decisive result: interfaces
1/3/4 are *bit-for-bit USB Audio Class 2.0* -- standard AC/AS subclasses (`0x01`/`0x02`),
UAC2 protocol (`bInterfaceProtocol 0x20 = IP_VERSION_02_00`), standard class-specific
descriptors, a standard UAC2 clock model, and standard UAC2 control requests. The *only*
non-standard byte is `bInterfaceClass = 0xFF` (vendor) instead of `0x01` (audio), which is
exactly what stops `snd-usb-audio` from binding. Digidesign shipped a UAC2 interface and
relabelled its class so only their driver would attach.

Confirmed topology and format (all from the config descriptor + live ISO stream):

[cols="1,4",options="header"]
|===
| Property | Confirmed value
| Playback (host->unit), intf 3 alt 1, EP `0x03` OUT | 6 ch, `S32_LE` (4-byte subslot, `bBitResolution` 32), ISO async, 416 B max/pkt, interval 1
| Capture (unit->host), intf 4 alt 1, EP `0x83` IN | 8 ch, `S32_LE` (4-byte subslot, `bBitResolution` 32), ISO async + *implicit feedback*, 416 B max/pkt, interval 1
| Bit depth | Format descriptor declares 32-bit in a 4-byte subslot, but the Eleven's converters are *24-bit* (Avid spec) -- 24 significant bits carried in 32-bit slots. ALSA will still expose `S32_LE`; the low bits are just not meaningful.
| Sample rates | Device supports *44.1 / 48 / 88.2 / 96 kHz*, all 24-bit, up to 8 ch (Avid spec). Only 44100 appears in the capture -- the editor `SET_CUR`s whichever rate the session uses. All four fit the 416 B packet (worst case 96 k x 8 ch x 4 B = 384 B/uframe). Rate is clock-programmable, not a format list.
| Clock | UAC2 clock selector `0x80` -> 3 sources: `0x81` internal programmable (freq control r/w; the four internal rates), `0x82`/`0x83` external -- the AES/EBU and S/PDIF digital inputs, which can slave the unit. Editor selects `0x81`.
| Sync model | Playback OUT is async and *slaved to the capture IN packet cadence* via implicit feedback (EP `0x83` `bmAttributes 0x25`). No separate feedback endpoint. `snd-usb-audio` handles this natively.
|===

The class-control bring-up the editor sends (interface 1, all standard UAC2) is:

  SET_CUR  bReq 0x01  wValue 0x0100  wIndex 0x8001  data 01            # CLOCK_SELECTOR_CONTROL(sel 0x80) = source 1
  SET_CUR  bReq 0x01  wValue 0x0100  wIndex 0x8101  data 44 AC 00 00   # SAM_FREQ_CONTROL(clock 0x81) = 44100
  SET_CUR  bReq 0x01  wValue 0x0100  wIndex 0x8001  data 01            # (re-assert selector)
  SET_CUR  bReq 0x03  wValue 0x0000  wIndex 0x2001  data 02            # extension unit 0x20 (Avid-specific; probably optional)

Only the last write touches a non-UAC2 concept (the `0x09` EXTENSION_UNIT id `0x20`); a
stock UAC2 driver ignores extension units and should still stream. Whether that write is
required for audio to flow is the one thing left to test on hardware.

=== What implementing a real soundcard driver now requires

Two viable paths; do Path A first -- it is a few lines and is decisive.

*Path A -- kernel quirk (recommended, least effort).* Add a `USB_DEVICE(0x0dba, 0xb011)`
composite quirk to `sound/usb/quirks-table.h` that forces `snd-usb-audio` to claim
interfaces 1/3/4 as `QUIRK_AUDIO_STANDARD_INTERFACE` (and leave interface 2 as
`QUIRK_MIDI_STANDARD_INTERFACE`, interface 0 to DFU). Because the descriptors are already
valid UAC2, the in-tree parser -- clocks, terminals, implicit feedback, rate setting --
should light up an ALSA `pcmC2D0` with 8 in / 6 out at 44.1 kHz (and whatever else the
clock's `RANGE` reports) with no format RE at all. This is the standard fix for
"UAC-standard device mislabelled vendor-class." Risk: the parser may reject an
Avid-specific quirk in an entity descriptor, or the unit needs the extension-unit write;
both are quick to diagnose with `dmesg`. A third, rate-specific risk: `snd-usb-audio`
enumerates the supported rates by issuing UAC2 `GET RANGE` on clock `0x81`, but the
capture shows Avid's own driver *only ever `SET_CUR`s* the rate and never queries `RANGE`
-- so whether the firmware answers `RANGE` at all is untested. If it stalls/errors, expect
only 44.1 kHz (or a clock-init failure) and reach for a fixed-rate / rate-list quirk. All
four rates (44.1/48/88.2/96) fit the endpoint, so this is a firmware-response question,
not a bandwidth one. Deliverable: a kernel patch against Linux 6.18,
then upstream to alsa-devel if it holds. The patch (`eleven-rack-uac2-quirk.patch`), a
NixOS module that applies it (`nixos-eleven-rack.nix`, pins kernel 6.18), the entry
source, a regenerator script, and the build/test/verify recipe are in
`contrib/eleven-rack-alsa-quirk/`.

*Path B -- userspace driver (fits this repo's libusb philosophy).* Grow
`eleven-audio-probe` into a real UAC2 client: query the clock `RANGE`, `SET_CUR` the rate,
select alt 1 on both AS interfaces, run async ISO OUT whose submission rate tracks the IN
endpoint's implicit feedback, format real 32-bit PCM, and bridge to PipeWire/JACK (or an
ALSA loopback). More work than Path A but needs no kernel patch and is distro-portable.
Use `nusb` (pure-Rust async ISO) rather than fighting rusb's missing ISO API. The channel
map (which of the 6 playback / 8 capture channels are guitar / dry / reamp / USB-return)
still needs labelling from the terminal descriptors + listening tests, but that is cosmetic
routing, not a blocker to first sound.

== RESOLVED (2026-07-16): working driver

The driver is *working* on real hardware (kernel 6.18.38): 8-ch capture / 6-ch
playback, 24-bit, 44.1/48/88.2/96 kHz, full-duplex, zero xruns; mixer + MIDI too.
The final fix (in `contrib/eleven-rack-alsa-quirk/eleven-rack-uac2-quirk.patch`)
addresses three device non-conformances across five files:

* *quirks-table.h* -- composite quirk forcing standard UAC2 parsing on the
  vendor-class (0xFF) interfaces (mixer/MIDI/streaming).
* *card.c* -- pick a real AudioControl interface for `chip->ctrl_intf` (the DFU
  interface 0 enumerates first and otherwise breaks terminal/clock lookups).
* *format.c* -- fixed rate-list fallback (the firmware does not answer SAM_FREQ
  RANGE).
* *pcm.c* + *quirks.c* -- `QUIRK_FLAG_IGNORE_CTL_ERROR` + non-fatal PITCH enable
  (the endpoints declare a read-only pitch control the firmware NAKs).

The sections below record the earlier (superseded) attempts and the RE that led here.

=== Lab validation (2026-07-16): scope + signal generator

Validated end-to-end with instruments (Rigol DG1062Z generator on the guitar
input, Rigol MSO4014 scope on the headphone/Main output), driven over raw-TCP
SCPI:

* *Capture is pitch-accurate.* A 1 kHz sine into the guitar jack is captured at
  ~1000 Hz on ch1 and ch3 (zero-crossing), with exact per-second frame counts.
* *Playback is clean.* A 1 kHz tone played over USB reads on the scope as a
  clean sine: frequency 999 Hz (sd 7), Vpp 464 mV (sd 5), stable Vrms, 0
  dropouts over the run. So the driver clocks playback at exactly 48 kHz and
  produces a correct waveform.
* *Channel map (confirmed):*
  capture ch1 = Guitar In (dry DI), ch2 = Mic In, ch3-4 = Eleven Rig L/R (wet),
  ch5-6 = Digital In L/R, ch7-8 = Line In L/R;
  playback ch1-2 = Main Out L/R (mirrored to Phones), ch3-4 = ReAmp L/R,
  ch5-6 = Digital Out L/R. Guitar (ch1, dry) and Rig (ch3-4, wet) are the same
  source at two taps -- the dry/wet pair for modelling.

*Two red herrings, resolved by the scope:* the earlier "playback runs 1.5x slow
/ is really 4-channel" and "choppy/garbled" conclusions were artifacts, not
driver bugs. The unit is on a QEMU virtual XHCI (USB passthrough), which forwards
isochronous IN (capture) faithfully but backpressures/jitters isochronous OUT
(playback), polluting `aplay` wall-clock timing; and several scope reads landed
on the silence right after a test tone ended. The scope's correct-frequency
result disproves the rate/channel theories -- the device is genuinely 6-channel
playback at 48 kHz. Expect clean playback on bare metal.

*Userspace note:* the kernel driver is validated; the PipeWire path still needs
tuning (an async device that PipeWire must resample; node volume defaulted to 0;
`pw-play` routing). `contrib/eleven-rack-alsa-quirk/wireplumber-eleven-rack.{conf,nix}`
is a baseline (no node suspend so the clock lock holds; Main as FL/FR, rest
discrete AUX; the channel map above).

=== Earlier attempts (superseded): quirk binds; v1 no PCM; v2 no stream

Path A was built (patch against 6.18, deployed on a NixOS test VM, kernel 6.18.38)
and tested on the real unit.

* *v1 (`QUIRK_DATA_STANDARD_AUDIO` on interfaces 3/4)*: the card, `usbmixer`, and MIDI
  come up, but *no PCM* -- both streaming interfaces are rejected:
  `3:1 : bogus bTerminalLink 0`, `4:1 : bogus bTerminalLink 1`, plus
  `64:1: cannot get min/max values for control 7..14`.
* *Root cause (confirmed against 6.18 source)*: the DFU interface (0) enumerates first
  and becomes `chip->ctrl_intf`. The composite quirk's `STANDARD_AUDIO` path calls
  `snd_usb_parse_audio_interface()` directly *without* registering a per-interface
  control link, so `snd_usb_find_ctrl_interface()` falls back to the DFU interface for
  interfaces 3/4. Terminal-link *and* clock resolution (both use that interface's
  descriptors) then fail; the mixer `min/max` errors are the same misdirection. The
  `sound/usb/card.c` comment at the `chip->ctrl_intf` assignment literally anticipates
  this ("we might need a more specific check here in the future").
* *v2 (fixed-endpoint `QUIRK_AUDIO_FIXED_ENDPOINT`)*: describe the two streams with
  explicit audioformats so `create_fixed_stream_quirk()` builds the PCMs directly,
  skipping terminal parsing; generic implicit-feedback detection links playback to the
  capture interface. This is the candidate fix (in `contrib/eleven-rack-alsa-quirk/`),
  pending the next hardware build. Residual: rate `SET_CUR` still routes through
  `ctrl_intf`, so non-default rates may not take -- set the rate on the unit's front
  panel until an upstream `ctrl_intf` fix lands. The full-fix option is a small kernel
  patch making `ctrl_intf` resolve to the real AudioControl interface (1).

== Why ALSA gives us nothing today

The Eleven exposes 5 USB interfaces (config has 5, all in one configuration):

[cols="1,3,3",options="header"]
|===
| Intf | Class / subclass | Role
| 0 | `0xFE` AppSpec / AudioControl | audio-control header (app-specific)
| 1 | `0xFF` VENDOR / AudioControl, proto 32 | vendor audio-control topology (terminals/feature units in CS descriptors)
| 2 | `0x01` Audio / *MIDIStreaming* | **standard USB-MIDI** — bulk EP `0x02` OUT / `0x82` IN. This is `hw:2,0`, what `rackctl-eleven` already uses.
| 3 | `0xFF` VENDOR / AudioStreaming, proto 32 | **audio playback (host→unit)**. alt 0 = idle (no EP); alt 1 = ISO EP `0x03` OUT, 416 B/pkt, interval 1.
| 4 | `0xFF` VENDOR / AudioStreaming, proto 32 | **audio capture (unit→host)**. alt 0 = idle; alt 1 = ISO EP `0x83` IN, 416 B/pkt, interval 1.
|===

Interfaces 3 and 4 are **vendor class (`0xFF`)**, not UAC (`0x01`/AudioStreaming).
The kernel `snd-usb-audio` driver only binds standard UAC audio, so it ignores 3/4 —
which is why card 2 has `midiC2D0` but **no `pcmC2*`**. Nothing on Linux ever opens
the audio path.

Audio format (from the alt-1 CS format descriptors, both directions):
`06 24 02 01 04 20` = TYPE_I PCM, subframe `0x04` (4 bytes), `0x20` = 32-bit; ISO
416 B *max*/packet at interval 1 (8000 packets/s at high speed); the live stream runs
~176 B/pkt (capture) and ~132 B/pkt (playback) at 44.1 kHz. Playback AS links to input
terminal `0x00`, capture AS to output terminal `0x01`. Channel count and clock are now
CONFIRMED -- see "KEY FINDING" above: 6-ch playback, 8-ch capture, 44100 Hz, UAC2 clock.

== Approach: libusb userspace driver (no kernel work)

Because 3/4 are vendor class with **no kernel driver attached**, a userspace libusb
program can claim them directly and coexist with MIDI (which stays on ALSA `hw:2,0`,
interface 2 — a different interface, so no conflict). No kernel module, no
`snd-usb-audio` quirk patch needed for the feasibility phase.

Sketch: open `0dba:b011` → `claim_interface(3)`, `claim_interface(4)` →
`set_interface_alt_setting(3,1)` and `(4,1)` → submit ISO transfers to `0x03` (OUT)
and `0x83` (IN) on a steady cadence. Keep it running as a side process while the
existing MIDI tooling talks to `hw:2,0`.

== The feasibility gate (cheap, decisive — do this FIRST)

The whole project pivots on *how much* audio the edit-gate actually needs. Test three
escalating hypotheses, cheapest first, each a small addition to the libusb sketch.
After each, send a known MIDI param-write and read it back (or diff the captured
blob) to see if the write now *sticks*:

[cols="1,3,2",options="header"]
|===
| H | What we activate | Effort if this is enough
| H1 | `SET_INTERFACE(3,1)`+`(4,1)` only — interface active, no ISO data | ~½ day
| H2 | H1 + submit ISO OUT/IN URBs with *silence / dummy* payloads at the right cadence | ~2–3 days
| H3 | H2 but payloads must be *correctly formatted* PCM + honour the sync/clock | weeks (full RE)
|===

If **H1 or H2** opens the edit gate, we get parameter editing for a few days' work
and *don't need to decode the audio content at all*. Only H3 requires the full
proprietary-format reverse-engineering. This test tells us which world we're in
before committing.

== Effort tiers & risks

* *Tier A — edit-gate only (H1/H2):* small libusb activator, run beside the GUI's
  MIDI. Unlocks the parameter editor if the theory holds. Main risk: the gate needs
  H3 (real audio), or needs the *cold-connect* control sequence we don't yet have
  (our captures start mid-session — a fresh plug-in + editor-launch capture would
  fill that gap).
* *Tier B — working audio I/O (H3):* decode the ISO PCM format, channel map, and
  feedback/clock; expose it as an ALSA loopback or JACK/PipeWire client. Weeks;
  independent value (monitoring) even if editing needs more.
* Risks: proprietary sync model; the 416 B/pkt cadence and any explicit-feedback
  endpoint; high-speed ISO scheduling from userspace (libusb async ISO); and the
  unknown cold-init control transfers (need a plug-in capture to see them).

== The Windows activation sequence (from the cold-connect capture)

Decoded from `11/eleven-save-20260702-114348.pcapng` (power-cycle → capture →
launch editor). At t≈14 s the editor brings audio up with exactly:

  SET_CONFIGURATION(1)
  SET_INTERFACE(3,0) (4,0)          # idle reset (sent twice)
  SET_INTERFACE(3,1) (4,1)          # activate the ISO endpoints
  # then class SET_CUR writes (bmRequestType 0x21) to interface 1:
  bReq 1  wValue 0x0100 wIndex 0x8001  data 01            # entity 0x80 enable?
  bReq 1  wValue 0x0100 wIndex 0x8101  data 44 AC 00 00   # 0xAC44 = 44100 Hz
  bReq 1  wValue 0x0100 wIndex 0x8001  data 01
  bReq 3  wValue 0x0000 wIndex 0x2001  data 02            # entity 0x20

`eleven-audio-probe` replays all of this (minus `SET_CONFIGURATION`, which would
reset the device and drop the MIDI interface).

== Results so far

* **H1 tested → NOT sufficient.** The probe activates interfaces 3/4 (alt 1) and
  replays every class control write (all ACKed, sample rate set), *and* MIDI keeps
  working alongside it (a `scan amp` read succeeds during the hold — libusb audio +
  ALSA MIDI coexist cleanly). But a verbatim editor param-write still changes
  nothing (blob 0 differing bytes, amp table unchanged). So arming the endpoints +
  control setup is not enough.
* **H2 tested → also NOT sufficient.** The probe now streams ISO on EP `0x03`/`0x83`
  via `libusb1-sys` async transfers; it verifiably flows (≈12,000 clean completions
  in 6 s, only the shutdown cancels error). With silence streaming *and* audio
  active, the editor param-write STILL doesn't stick, and `01 11 78 0D …` reads
  return nothing — the same as with no audio, whereas the editor's session always
  gets those reads answered. So streaming silence audio does not open the param
  interface.
* *Reassessment:* the audio-session theory is weakened — a fully-flowing (silent)
  audio stream changes nothing. Either the gate needs **H3** (real, clock-locked
  PCM, not silence — much harder and uncertain), or it is **not audio at all** but a
  MIDI session-open we haven't identified. The doc note "the editor's session always
  gets `01 11` reads answered" points at a MIDI-side opener; the cold-connect
  capture's *MIDI* stream (not yet fully mined) is the next place to look, and is far
  cheaper than H3. The libusb ISO streamer built here is reusable for H3 / Linux
  audio regardless.

== Concrete next steps

. Implement **H2**: submit ISO transfers on EP `0x03` (OUT) and `0x83` (IN). rusb has
  no ISO API, so use `nusb` (pure-Rust async ISO — preferred, no `unsafe`) or
  `libusb1-sys` async. Start with silence (all-zero 416-byte packets) at 8 pkts/ms.
. Re-run the edit-gate test while H2 streams. If it opens → **Tier A done**: wire the
  activator into `eleven-gui` so editing works while it holds the session.
. If silence isn't enough, escalate to correctly-clocked / formatted PCM (**H3**,
  Tier B — the full driver), which also yields Linux audio monitoring.

Until this lands, the GUI stays at bank-level editing (capture / restore / copy /
rename / scene — all working). See `eleven-rack-sysex-protocol.adoc` for the (now
known) host-set message format that this session would make effective.
