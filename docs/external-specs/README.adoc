= External specifications
:icons: font

Third-party reference documents used while implementing the device crates. The
PDF binaries are *not* committed (they are vendor-copyrighted and this is a
public repository); they are git-ignored. This manifest records what each one is
and where to fetch it, so the reference is reproducible without redistributing
copyrighted material.

To restore the local copies, download each `URL` into this directory and check
the SHA-256.

== Documents

| File | Source | Notes

| `GX-700_MI.pdf`
| https://cdn.roland.com/assets/media/pdf/GX-700_MI.pdf
| Roland *GX-700 MIDI Implementation* (#10799, dated Mar 8 1996, v1.00). The
  authoritative System Exclusive reference: the DT1/RQ1 message formats, the
  4-byte address map, and the full parameter tables (effect blocks, ranges, enum
  values). This is the document Roland's Owner's Manual refers to as the optional
  "MIDI Implementation booklet". Our reverse-engineered understanding is written
  up in `../gx700-sysex-protocol.adoc` and encoded in `crates/gx700`. +
  SHA-256: `ecc3527ab2463a69cea1964970a1aa8dc196194e54f0b87c308a956dcf2e9fad`

| `GX-700_OM.pdf`
| https://cdn.roland.com/assets/media/pdf/GX-700_OM.pdf
| Roland *GX-700 Owner's Manual* (72 pp, scanned). General operation, the
  one-page MIDI Implementation Chart, and the bulk dump/load procedures. Confirms
  the factory device id (1 -> protocol `0x00`) and the effect-chain model. The
  "Receiving Data (Bulk Load)" procedure (pp. 51-54) is the source of the BULK LOAD
  requirement for storing patches over MIDI. +
  SHA-256: `a1db96487bdb4f5f29cf5bb832ccea2ba081cb8d5f1f0847c06de7c504940293`

| `Boss-Gx-700-Service-Notes.pdf`
| https://www.manualslib.com/manual/3677678/Boss-Gx-700.html (also on manuals.plus
  / Scribd; no Roland CDN copy)
| BOSS *GX-700 Service Notes* (29 pp, scanned, Feb 1996). A hardware service manual
  -- parts list, block diagram, board layouts. Useful confirmations: the front-panel
  *test mode* ([PREAMP]+[SP.SIM] on power-up) whose only MIDI item is a hardware
  IN/OUT loopback test (no remote command), and the *FACTORY INITIALIZE* procedure
  (UTILITY mode) that resets the System parameters and user patches P#1-P#100 to the
  factory presets. No new MIDI/SysEx protocol detail beyond `GX-700_MI.pdf`. +
  SHA-256: `cde5f8b8dccd828aa57dba3361c7302dfbcf03f1a8e1b758a86f70a5ee93ad0f`

| `Eleven_Rack_User_Guide.pdf`
| https://www.soundland.de/media/pdf/e1/b0/f4/Eleven_Rack_User_Guide_v804_66715.pdf
  (also on Avid's KB, article en358799)
| Avid/Digidesign *Eleven Rack User Guide* (148 pp, v8.0.4). Operation manual for
  the third device. *Chapter 11, "Controlling Eleven Rack with MIDI"* is the useful
  part: a *MIDI CC reference* listing, per amp model and per effect, which Continuous
  Controller drives each parameter (and the block *bypass* CCs: DIST 25, MOD 50, DELAY
  28, REVERB 36, FX LOOP 107, WAH 43, AMP 111, ...), plus the *Program Change* /
  User-Factory *Bank Change* rig-select scheme. This gives the parameter *names,
  per-amp structure and value semantics* that help label our reverse-engineered
  addresses, and the rig-select mapping. It does *not* document the editor's
  address-mapped System Exclusive protocol (that is reverse-engineered in
  `../eleven-rack-sysex-protocol.adoc`; the rig file format in
  `../eleven-rack-rig-format.adoc`). +
  SHA-256: `e6bb6e8554d09aabe46f6542c42b0717b3b1cb00da952c47abbefca3cabffb5b`

| `user-manual-30336_(manymanuals.com).pdf`
| https://toneprints.com/media/216833/tc_electronic_g-system_manual_english.pdf
  (official TC English manual; this local copy came via manymanuals.com doc #30336)
| TC Electronic *G-System User's Manual* (v2.04, 81 pp). The G-System's rack unit is
  model *GFX01*; a candidate device for a future `g-system` crate. The MIDI menu and
  the one-page *MIDI Implementation Chart (v1.0)* are the useful part, and they show a
  *preset-recall + bulk-backup only* MIDI model: it responds to *Program Change* (with
  *CC#0* bank-select MSB -- value 0 = 100 factory, 1 = user 1-100, 2 = user 101-200 --
  then PC 0-99) to recall presets; it *sends* PC out (plus an optional remappable Prg.
  Map) and lets footswitches transmit assignable *CC 0-119*; *SysEx is bulk-dump only*
  (Utility -> Dump System / Dump Bank, `SysEx ID` 0-126) with *no per-parameter
  get/set*. Real-time parameter control is via *expression pedals wired locally to the
  GFX01* (default assign Input Gain / Volume), not over MIDI. 100 factory + 200 user
  presets. Pitch Bend / Notes / Aftertouch unsupported. +
  SHA-256: `cefa203b78223b8b910a4dce992595114bf2587efd7254e2a0b7345daa0db64e`

| `G-SYSTEM-Tips.pdf`
| Psicraft *Vyzor G-System* editor/librarian, distributed with the software (TC
  Electronic). Filename is misleading -- it is the Vyzor guide, not a TC tips sheet.
| *Vyzor G-System User's Guide* (Psicraft, 25 pp). Despite the local filename this is
  the manual for a *third-party editor/librarian*, and it is significant because it is
  proof that *full per-parameter editing of the G-System over MIDI is possible* --
  contradicting the limited picture in the v2.04 user manual above. Vyzor *GET*s all
  data from the unit over MIDI In/Out, edits *every preset parameter in real-time* into
  the G-System's temporary edit buffer, and *SYNC*s presets/banks back. It requires
  G-System firmware *>= 4.02 R1737* (newer than the v2.04 manual's chart), i.e. the
  deep-edit MIDI capability arrived in later firmware. The editor's parameter groups
  (*Filter, Dyn|EQ, Mod, Pitch, Delay, Reverb, Modifier*) map the preset structure.
  Vyzor is *not* G-System-specific: it is a generic Psicraft editor/librarian engine
  skinned per device via a device profile/mapping (same engine as *Vyzor Evolver* for
  the DSI Evolver and the *Vyzex* line for Line 6 PODs, etc.), so the G-System parameter
  map lives in a per-device profile plus parseable user-data files (`.SQS` = a full
  device backup Set, `.SQC` = the whole Collection). The wire protocol is *undocumented*
  (proprietary), so a `g-system` editor crate would require *reverse engineering* --
  either extracting that device profile or parsing `.SQS`/`.SQC` dumps, then validating
  against MIDI traffic, the same approach used for the Eleven Rack. (Note: Vyzor's "CC
  Learn" maps control-surface CCs to editor controls; that is a host feature, not a
  G-System protocol capability.) +
  SHA-256: `02e3d039ea0d805b8f20c1746ca3e40c00926a86fa68042dd433d77b2d0c36c8`
